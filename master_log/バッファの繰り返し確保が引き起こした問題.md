## 背景

パノラマ動画（120フレーム）から撮影画像に最も類似するフレームを探す位置同定システムをMPIクラスタ上で実装した。  
13台のMac mini（ワーカーノード）を使い並列化したにもかかわらず、**1ワーカーあたりの処理時間が約20秒**かかるという問題が発生した。

```
システム構成:
  マスターノード  : Mac Studio (Rank 0) × 1台
  ワーカーノード  : Mac mini   (Rank 1〜13) × 13台
  総MPIプロセス数 : 14 (-n 14)
  OpenCL        : pyopencl (CPU ベース OpenCL)
  タスク         : 109フレーム分の SIFT 128次元記述子をマッチング
```

## GPUバッファとは

OpenCL では CPU メモリと GPU メモリは**別の空間**にある。  
CPU 上のデータを GPU カーネル（計算プログラム）に渡すには、まず **GPU 側にメモリ領域（バッファ）を確保**し、データをコピーする必要がある。

```
CPU メモリ                         GPU メモリ
┌─────────────────┐                ┌──────────────────┐
│  特徴点データ    │  ──HtoD転送──▶ │  cl.Buffer       │
│  (numpy array)  │                │  (GPU バッファ)   │
└─────────────────┘                └──────────────────┘
                                           │
                                      カーネル実行
                                    （SIFTマッチング）
                                           │
┌─────────────────┐                ┌──────────────────┐
│  結果 (距離値)   │  ◀─DtoH転送── │  結果バッファ     │
└─────────────────┘                └──────────────────┘
```

| 用語 | 意味 |
|------|------|
| `cl.Buffer` | GPU 側のメモリ領域。`malloc` に相当 |
| HtoD (Host to Device) | CPU → GPU へのデータ転送 |
| DtoH (Device to Host) | GPU → CPU への結果転送 |
| ブロッキング同期 | DtoH 完了まで CPU が**待機**すること |

## 発生していた問題

### 症状

| 項目 | 値 |
|------|-----|
| GPU カーネルの実際の実行時間 | **0.289 秒** |
| 実測の処理時間 (wall time) | **約 20 秒** |
| オーバーヘッド倍率 | **約 70 倍** |

「計算自体は 0.3 秒で終わっているのに、なぜ 20 秒かかるのか」が問題だった。

### 旧コードの処理フロー

`LOCAL_BATCH_MAX=4`（4フレームずつ処理）の場合、109フレームの処理で以下を **28回繰り返していた**。

```
┌─── 1サイクル (これを 28回繰り返す) ───────────────────────────────┐
│                                                                    │
│  ① cl.Buffer 確保   ← GPU メモリを新規確保（malloc相当）           │
│  ② HtoD 転送        ← 4フレーム分のデータを GPU へコピー           │
│  ③ カーネル実行      ← SIFTマッチング計算（実際の処理）             │
│  ④ DtoH 転送        ← 結果を CPU へコピー ★ここでブロッキング待機  │
│  ⑤ cl.Buffer 解放   ← GPU メモリを解放（free相当）                │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

### 時間の内訳（計測値）

```
1サイクルあたり:
  ③ カーネル実行       ≈  0.010 s  ( 2.3%)  ← 実際の計算
  ④ ブロッキング待機   ≈  0.440 s  (97.7%)  ← ほぼ待つだけ

28サイクルの合計:
  カーネル実行の累積   ≈  0.289 s
  ブロッキング待機の累積 ≈ 12.3 s
  ──────────────────────────────
  合計 wall time       ≈ 20 s
```

**ブロッキング DtoH 待機（1回 ≈ 0.44秒）が 28回繰り返されることで、12 秒以上の待機時間が蓄積していた。**

> **なぜブロッキング待機が長いのか？**  
> DtoH 転送は「GPU がカーネルを完全に終了し、結果をバッファに書き出す」まで待つ。  
> CPU-based OpenCL（Apple Silicon の場合）では、このキュー同期コストが 1回あたり約 0.44 秒かかる。

---

## 原因の特定

OpenCL のイベントプロファイリング（`PROFILING_ENABLE`）を使い、各処理ステップの実 GPU 時間を計測した。

```python
# プロファイリング有効なキューを作成
queue = cl.CommandQueue(
    ctx,
    properties=cl.command_queue_properties.PROFILING_ENABLE
)

# イベントを取得して実行時間を計測
ev_kernel = kernel(queue, (global_size,), (local_size,), ...)
ev_dtoh   = cl.enqueue_copy(queue, result, buf_min1, is_blocking=True)

ns = 1e-9
kernel_time = (ev_kernel.profile.end - ev_kernel.profile.start) * ns
```

### 計測結果（20秒版, Rank 1）

```
GPU Wall time (全体)    12.597 s   100.0%
├── host_overhead       12.307 s    97.7%  ← ブロッキング待機の累積
└── GPU event_total      0.290 s     2.3%  ← 実際の GPU 計算
    ├── HtoD (転送)      0.000 s     0.0%
    ├── Kernel            0.289 s     2.3%
    └── DtoH (転送)      0.000 s     0.0%
```

**GPU は 2.3% の時間しか使われていなかった。**

---

## 解決方法

### 方針：**バッファをメイン処理の前に一括で事前確保し、使い回す**

処理のたびに確保・解放するのをやめ、**起動時に最大サイズ分のバッファを 1 回だけ確保**（プリアロケーション）し、毎回のマッチング処理ではそのバッファに上書きする形に変更した。

```python
# ========== 変更前（旧コード）==========
# マッチングのたびに確保・解放（28回繰り返される）
for batch in batches:
    buf = cl.Buffer(ctx, mf.READ_ONLY, size=batch_size)  # 毎回確保
    cl.enqueue_copy(queue, buf, data, is_blocking=False)
    kernel(queue, ...)
    cl.enqueue_copy(queue, result, buf, is_blocking=True) # ブロッキング
    buf.release()                                         # 毎回解放


# ========== 変更後（新コード）==========
# 初期化時に 1 回だけ確保（メイン処理の前）
def init_opencl(max_N=20000):
    buf_pano = cl.Buffer(ctx, mf.READ_ONLY,  size=max_N * 131 * 4)  # 事前確保
    buf_min1 = cl.Buffer(ctx, mf.WRITE_ONLY, size=max_N * 4)        # 事前確保
    buf_min2 = cl.Buffer(ctx, mf.WRITE_ONLY, size=max_N * 4)        # 事前確保
    return buf_pano, buf_min1, buf_min2  # ← 使い回す

# マッチング処理では上書きするだけ（確保・解放なし）
# 全フレームを 1 バッチで処理（LOCAL_BATCH_MAX=200）
cl.enqueue_copy(queue, buf_pano, all_data, is_blocking=False)  # 上書き
kernel(queue, ...)
cl.enqueue_copy(queue, result, buf_min1, is_blocking=True)     # 1回だけ
```

### 変更のポイント

| 項目 | 変更前 | 変更後 |
|------|--------|--------|
| バッファ確保のタイミング | マッチングのたびに動的確保 | **起動時に 1 回のみ事前確保** |
| バッファの使い方 | 確保 → 使用 → 解放を繰り返す | **上書き → 使用** を繰り返す |
| カーネル呼び出し回数 | 28 回（4フレームずつ） | **1 回**（全フレーム一括） |
| ブロッキング同期の回数 | **28 回** | **1 回** |
| この手法の名称 | 動的確保 | **プリアロケーション（事前確保）** |

## 計測結果の比較

### 1ワーカーあたりの GPU 時間内訳

| | 20秒版（旧）| 0.5秒版（新）|
|--|------------|-------------|
| GPU Wall time | 12.597 s | 0.526 s |
| host_overhead | 12.307 s (97.7%) | 0.513 s (97.6%) |
| GPU event_total | 0.290 s (2.3%) | 0.012 s (2.4%) |
| カーネル呼び出し回数 | 28 回 | 1 回 |

> host_overhead の**割合は同じ（≈97.6%）**。  
> 改善後も 1 回のブロッキング待機（0.44s）は発生するが、**28回 → 1回**になったことで累積コストが激減した。

### システム全体のスループット比較

|------|---------|---------|------------|

## まとめ

```
問題:  GPU バッファを処理のたびに確保・解放していた
         → ブロッキング同期コスト（0.44s）が 28 回蓄積
         → GPU 使用率 2.3%、残り 97.7% は CPU が待機するだけ
         → 実際の計算 0.29s に対して wall time が 20s

解決:  メイン処理の前に最大サイズ分のバッファを 1 回だけ事前確保
         → カーネル呼び出しを 28 回 → 1 回に集約
         → ブロッキング同期も 1 回のみ
         → wall time が 20s → 0.5s（40倍高速化）
```

**GPU を使う処理では「バッファの確保・解放は高コスト」であり、可能な限り事前に確保して使い回す設計が重要。**
